# 部署前檢查清單（防止「誤改未計畫項目」）

> 背景：曾經發生「做某項更新時，順手改到了原本沒打算改的內容」（例如官網器材／知識頁面、章別描述被截短、私人訓練只在官網改而其他家長端沒跟住改）。
> 以下流程令同類問題唔再發生。**每次 push 上 `main`（會自動部署）之前，逐項做齊。**

## 0. 先講清楚「今次計畫要改乜」
- 用一兩句寫低今次改動範圍（邊個檔、邊個功能）。之後所有 diff 都要對得返呢個範圍。

## 1. 跑自動一致性檢查
```bash
node tools/check-consistency.mjs
```
- 任何一項 ✗ → **停手**，唔好部署，先查係咪計畫內。
- 涵蓋：私人訓練(PT)三檔內容一致、PT 無「張Sir」、家長端有 PT 按鈕、章別 Level 1–14 齊全、官網器材清單核心項目存在。

## 2. 全量 diff 逐塊分類
```bash
git diff --stat            # 邊啲檔被改
git diff                   # 逐行睇
```
- **每一段改動**只可以係兩類其一：
  - ✅ **計畫內**：對得返第 0 步寫低嘅範圍。
  - ❌ **計畫外**：唔喺範圍内 → **還原返**（`git checkout -- <檔>` 或手動改返），唔好順手帶住部署。
- 特別留意「內容陣列」（器材 `EQUIPMENT_ITEMS`、知識 `knowledge`、章別 `BADGE_DATA`、課程）有冇被無故改動／截短。

## 3. 「以官網為準」的共用內容
- 私人訓練(PT)內容嘅唯一真相 = `index.html (官網首頁, 舊 is-home.html 轉址)`（官網）。
- 若 PT 內容有改 → **三個檔一齊改**：`index.html (官網首頁, 舊 is-home.html 轉址)`、`is-parent.html`、`is-leave-makeup.html`，改完第 1 步嘅 PT 一致性檢查要 ✓。

## 4. 後端（apps-script）改動
- 涉及名額／收費／待補(owed)／繳費邏輯 → 部署前先同老闆確認。
- 重要資料寫入前要有 `backup()`，唔好令任何資料遺失。

## 5. 確認先部署
- 以上全部 ✓，而且老闆講咗「落D／上線／可以」先 push。
- Push 上 `main`：前端 HTML 由 GitHub Pages 自動上線；`apps-script/**` 由 GitHub Actions 自動 `clasp push` + `deploy`。

---
**一句總結**：先寫低今次要改乜 → 跑 `check-consistency.mjs` → 全量 diff 逐塊分類，計畫外一律還原 → 共用內容三檔一致 → 老闆 OK 先 push。
